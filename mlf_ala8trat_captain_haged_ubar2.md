# ملف اقتراحات الكابتن (v2)

> الهدف: مميزات جديدة للكابتن مبنية على نظامك الحالي (Real-time + Trips + Pending Requests + Safety + Pickup Hubs/Suggestions)، وتدي تميّز قوي لتطبيقك.

## 1) مشاركة موقع الراكب فورًا (Live Share Location)

**الفكرة**
- أول ما الراكب يفعّل **Share Location**، يبدأ إرسال تحديثات موقع الركوب (pickup) بشكل مباشر.
- الكابتن يشوف تحديثات نقطة الركوب Live في نفس اللحظة (بدون انتظار/تأخير)، وتتحدث الخريطة/المسار تلقائيًا.

**لماذا مميزة عندك؟**
- عندك أصلًا `PATCH /api/trips/:id/pickup` + متابعة GPS في الواجهة، لكن ينقصها “Real-time broadcast” للكابتن.
- إضافة بث Socket.IO لحدث pickup live تخلي التجربة “فورية” وتقلل مكالمات/رسائل.

**خطة تنفيذ (مختصرة)**
- Backend: عند تحديث pickup في `PATCH /api/trips/:id/pickup` يتم بث event للـtrip room مثل: `trip_pickup_live_update`.
- Frontend (Passenger): بدء `startPassengerPickupLiveUpdates(tripId)` فور تفعيل المشاركة + إرسال تحديث فوري (بدون انتظار أول interval).
- Frontend (Driver): استقبال event وتحديث marker/route مباشرة.

**مخرجات UX للكابتن**
- Badge صغيرة: "📍 الراكب يشارك موقعه الآن" + آخر دقة GPS (accuracy) وتوقيت آخر تحديث.

## 2) Driving Coach للكابتن (GPS + حساسات الموبايل)

**الفكرة**
- أثناء الرحلة (أو في وضع تدريبي)، يتم التقاط إشارات من حساسات الموبايل (تسارع/فرملة/انعطاف) لحساب “سلاسة القيادة”.
- نحفظ **ملخص فقط** (Counts/Score) بدل تخزين بيانات خام، لحماية الخصوصية وتقليل حجم البيانات.

**لماذا مميزة؟**
- تربط الأمان والربحية: قيادة أنعم = استهلاك أقل + تقييم أعلى + حوادث أقل.

**خطة تنفيذ (مختصرة)**
- Client: استخدام `DeviceMotionEvent` وحساب مؤشرات مثل: `hard_brake_count`, `hard_accel_count`, `hard_turn_count`.
- Backend: endpoint لتسجيل summary مرتبط بالرحلة.
- DB: جدول ملخصات القيادة (Trip-level summaries).
- Dashboard: شاشة/كارت للكابتن يوضح trend آخر 7 أيام + نصيحة واحدة قصيرة.

## 3) Incident / Dispute Package (باكدج أدلة جاهز)

**الفكرة**
- زر للكابتن: "فتح نزاع" أو "تسجيل حادث" للرحلة.
- النظام يجهّز تلقائيًا ملف أدلة من الموجود عندك:
  - إثبات انتظار (arrive/end)
  - تسجيل صوت مشفّر (لو موجود وبموافقة)
  - آخر مواقع + pickup accuracy + timestamps
  - Safety events (SOS/Help) إن وُجدت

**لماذا مميزة؟**
- بدل الشرح اليدوي والدوران بين شاشات، الأدلة تتجمع في “ملف واحد” سريع المراجعة.

**خطة تنفيذ (مختصرة)**
- DB: جداول `trip_disputes` + `trip_incident_packages` (أو جدول واحد يجمع الاثنين).
- API: إنشاء نزاع + جلب النزاعات + قرار إداري resolve.
- Admin UI: صفحة مراجعة نزاعات مرتبطة بالرحلات.

## قواعد التحقق قبل الدمج (حسب شرطك)

- اختبار الـAPI قبل أي commit: `npm test`
- اختبار build قبل أي commit: `npm run build`
- بعد نجاح الاختبارات: commit ثم push على `main`

> ملاحظة: لا يمكن ضمان 100% إن "ولا تطبيق تاني" عنده نفس الفكرة، لكن يمكن ضمان إن **التنفيذ المركّب على نظامك الحالي** (Socket.IO + أدلة الرحلة + privacy-first summaries) هو تميّز واضح لتطبيقك.
